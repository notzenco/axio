# Desktop UI audit — 2026-07-22

## Audit scope

Combined UX, visual-design, responsive, and screenshot-based accessibility
audit of the current Axio `0.0.1` interaction prototype. The audited journey
was: orient in a task, search/switch, create work, inspect agents, review a
diff, read command output, inspect a plan, adjust appearance, enter focus mode,
and use the workspace at compact widths.

The prototype uses deterministic demo data. This audit evaluates whether the
interface is a strong specification for the real repository, connector,
terminal, and Git work. It does not treat simulated behavior as implemented.

## User goal and accessibility target

A developer should be able to answer, without hunting:

1. What needs me now?
2. What is each agent doing?
3. Where will my next instruction go, with what authority?
4. What changed, what ran, and what remains?
5. What will Approve or Return actually do?

The target is a keyboard-usable desktop workflow with readable contrast,
non-colour state cues, predictable focus, safe error recovery, and useful
reflow down to the supported 720px native minimum width.

## Overall verdict

The shell is visually coherent and the task-first architecture is the right
foundation. It is not ready to become the implementation blueprint unchanged.
The next design pass should concentrate on truthful attention, safe task
creation, one canonical review surface, explicit composer context, and compact
overlay behavior. More decorative polish should wait until those contracts are
settled.

No P0 issue was observed. Seven P1 and seven P2 issues should be resolved in
design before substantial runtime implementation.

## Implementation follow-up — 2026-07-22

The audited prototype pass is implemented on `fix/desktop-ui-audit` against the
contract in [`ui-audit-implementation.md`](ui-audit-implementation.md). The
desktop shell is now a substantially stronger specification for runtime work:

- empty task submission stays in context with a persistent linked error;
- review status is explicit and decisions exist only in the Diff inspector;
- the inspector has changed-file navigation plus pinned and expanded layouts;
- compact panels are exclusive, scrim-backed, focus-contained, Escape
  dismissible, and restore focus;
- the composer exposes target, worktree, direction mode, and review policy;
- muted text contrast is at least 5.32:1 on the tested opaque surfaces;
- focus mode reports one consistent state and restores its prior layout;
- agent navigation is separate from explicitly labelled Pause/Resume controls;
- the command palette has New task and an announced no-results state;
- command output is presented as a completed record rather than an interactive
  terminal, and the plan is explicitly read-only with visible step states;
- routine history is calmer while evidence and attention retain boundaries.

Post-change evidence is recorded in [`../design-qa.md`](../design-qa.md). The
missing backend, failure, persistence, authentication, real Git, PTY, native
assistive-technology, 200% zoom, and high-contrast states listed below remain
deliberate pre-runtime work; this UI pass does not claim they exist.

## Strengths to preserve

- The task is the primary object. Timeline, composer, agents, and context all
  relate to the selected task rather than competing as separate applications.
- The timeline gives agent provenance a stable visual thread and pairs colour
  with names in the main narrative.
- The composer remains persistent while the task history scrolls.
- Local-only posture is visible in the titlebar, agent panel, and status bar.
- Native dialogs, semantic tabs, named icon buttons, and visible focus rings
  establish a useful accessibility foundation.
- The task view itself reflows cleanly at 720px without horizontal document
  overflow. The corrected native controls remain at the right window edge.
- Diff, output, and plan share one contextual surface rather than creating
  unrelated windows.

## P1 findings — resolve before implementation

### 1. Failed task creation destroys context instead of helping recovery

**Evidence:** Steps 3–4.

Submitting an empty Outcome closes the dialog and shows only a short-lived
toast at the bottom of the workspace. The user loses the form context and any
repository, worktree, or agent choices. The empty field is not marked inline.

**Design requirement:** Keep the dialog open, focus Outcome, show persistent
inline guidance, set `aria-invalid`, link the message with
`aria-describedby`, and announce it in a live region. Disable Create task until
the minimum valid state is present unless submission is needed to reveal a
server-side error. Preserve every entered choice after any recoverable failure.

### 2. Attention is duplicated but still not unmistakable

**Evidence:** Steps 1 and 6.

The same review gate appears as `Awaiting review`, a `Review 3` button, an
approval card in the timeline, and Return/Approve controls in the inspector.
The repetition creates uncertainty about whether these actions are equivalent,
while the tiny amber dot and small uppercase metadata still carry too much of
the urgency.

**Design requirement:** Define one task-level attention model and one canonical
action location. Use explicit labels such as `Review required` and focus the
exact attention event when invoked. Secondary mirrors may show status, but they
must not duplicate the decision controls.

### 3. The review inspector is not sufficient for real Git work

**Evidence:** Step 6.

The fixed-width pane immediately needs horizontal scrolling and displays one
file despite claiming three local changes. There is no file list, hunk
navigation, viewed state, binary/rename treatment, comment affordance, or clear
statement of what Approve changes. This would not safely support real review.

**Design requirement:** Specify peek, expanded, and pinned modes; add a changed
file list and keyboard hunk navigation; preserve task scroll position; and make
the consequence explicit (`Approve review`, `Return with feedback`, `Stage`,
`Commit`, and `Merge` must remain distinct operations).

### 4. Compact overlays compete for the whole window

**Evidence:** Step 11.

At the supported 720px width, the workspace sidebar and context inspector can
remain open simultaneously. They overlap the task and each other without a
scrim, obvious ownership, or mutual exclusion. The left panel obscures part of
the inspector while both titlebar toggles remain pressed.

**Design requirement:** Permit only one compact overlay at a time. Add a scrim,
focus containment, Escape dismissal, outside-click dismissal where safe, and
focus restoration to the invoking toggle. Opening one overlay must close the
other without changing task selection or scroll position.

### 5. The composer does not explain routing, context, or authority

**Evidence:** Steps 1, 6–8, and 12.

`All agents` is the only visible routing contract. A user cannot tell whether
the direction includes the selected diff, which worktree it targets, whether
it is an Ask/Direct/Plan interaction, or what permissions the agents receive.

**Design requirement:** Add a compact context strip for targets, task/worktree,
attachments or selected evidence, interaction mode, and permission level. The
send button should describe unusual scope, for example `Direct 2 agents`.

### 6. Small faint metadata misses the intended accessibility standard

**Evidence:** All desktop captures, especially Steps 2, 5, 7–9.

The `--faint` token (`#687181`) measures 4.02:1 against the deepest background
but approximately 3.35–3.68:1 on common opaque panel colours. It is frequently
used at 9–10px for timestamps, labels, commands, and status. This is below the
4.5:1 target for normal text and is visually difficult even without zoom.

**Design requirement:** Raise metadata contrast across every glass setting,
use larger optical sizes, and reserve very faint treatment for nonessential
decoration. Verify comfortable and compact modes at 200% zoom and Windows high
contrast. Do not make state depend on colour or tiny uppercase labels.

### 7. Focus mode exposes contradictory state

**Evidence:** Step 10.

The titlebar changes its button to the ambiguous label `Exit`, while the task
header still offers `Focus` and the context toggle remains pressed even though
the inspector is hidden. The interface therefore reports three different
stories about the same layout state.

**Design requirement:** Use `Exit focus mode`, hide or synchronize the duplicate
task action, and temporarily clear or visually suspend sidebar pressed states.
Restoring focus mode must return the prior panel state without claiming hidden
panels are currently visible.

## P2 findings — settle in the same design pass

### 8. Routine history is over-containerized

**Evidence:** Steps 1, 5–8, and 10.

Tool output, change summaries, and approvals all receive similarly large cards.
Routine history competes with the action that actually needs the user.

**Recommendation:** Use narrative rows, bounded evidence blocks, and attention
cards. Only questions, failures, conflicts, security gates, and approvals
should dominate.

### 9. Search has no empty-state or recovery path

**Evidence:** Steps 2 and 14.

No-match search leaves empty Tasks and Commands headings with no explanation,
clear action, or result count. Escape dismissal also could not be verified in
the browser harness.

**Recommendation:** Show `No tasks or commands match`, offer `Create task` or
clear-search actions where appropriate, announce result counts, and add an E2E
keyboard test for Escape and focus restoration.

### 10. Agent controls hide meaning and consequence

**Evidence:** Step 5.

Agent rows show only name and state, and clicking the row changes lifecycle
state immediately. `Session budget 18%` has no unit or definition. Provider,
worktree ownership, capabilities, current command, and recovery affordances are
absent.

**Recommendation:** Separate selection from lifecycle controls, label the
consequence (`Pause Codex`, `Resume Claude Code`), explain budget units, and
surface connector/worktree ownership plus failure recovery.

### 11. Command output is a snapshot, not a terminal contract

**Evidence:** Step 7.

The pane has no command history, timestamps, running/exit state, copy, clear,
follow-tail, stop, input, or wrapping controls. The small monospace text is hard
to scan.

**Recommendation:** Design explicit process states and a terminal toolbar
before building PTY integration. Keep raw output selectable and distinguish
historical evidence from an interactive terminal.

### 12. The plan is readable but operationally passive

**Evidence:** Step 8.

The plan communicates 4/6 progress but not ownership, blockers, timestamps,
source, editability, or why step 5 is current. There is no relation between a
plan item and its timeline evidence.

**Recommendation:** Define plan provenance and state vocabulary, link steps to
events, and decide whether users can edit, reorder, approve, or only observe a
provider-owned plan.

### 13. Compact titlebar semantics degrade

**Evidence:** Steps 11–13.

At 720px the focus button loses its text and its accessible name becomes the
icon glyph. The titlebar allocates substantial width to native controls and
leaves the remaining actions dependent on icons.

**Recommendation:** Preserve explicit `aria-label` values when visual labels
hide, keep tooltips, verify 40px targets where practical, and document the
priority order for hiding project, appearance, focus, and command details.

### 14. Glass and glow are applied too uniformly

**Evidence:** Steps 1, 5–10.

The shell is attractive but major surfaces share similar dark translucency,
borders, and elevation. Content hierarchy depends on small type and colour
rather than meaningful changes in surface treatment.

**Recommendation:** Keep glass for frame, navigation, composer, and transient
overlays. Give the timeline a calmer opaque reading surface and reserve glow
for current activity or required attention.

## Missing states that must be designed before real coding

The current demo cannot support a complete audit of the following. They should
be designed as matched happy, empty, loading, permission, error, and recovery
states before their runtime work begins:

- first launch, open repository, recent repositories, and invalid path;
- no task selected, no tasks, no agents, and returning-user overview;
- connector discovery, authentication required, launch failure, crash, stop,
  forced stop, and reconnect;
- permission request, user question, timeout, cancellation, and partial result;
- terminal streaming, long output, input, disconnect, and non-zero exit;
- real diff file navigation, large/binary/renamed files, dirty worktree,
  conflict, and external-tool handoff;
- persistence, restart reconciliation, stale worktree, missing branch, and
  corrupted local state;
- screen-reader announcements, keyboard-only completion, 200% zoom, high
  contrast, reduced motion, and native Windows focus behavior.

## Recommended pre-coding design sequence

1. **State contract:** Define task, agent, process, review, and attention states
   with exact labels, triggers, consequences, and recovery actions.
2. **Core journey wireframes:** Produce matched states for open repository,
   create task, launch one agent, steer it, answer a question, review a real
   diff, and recover from one failure.
3. **Hierarchy system:** Establish narrative row, evidence block, attention
   card, composer context strip, and inspector peek/expand/pin primitives.
4. **Responsive contract:** Specify wide, medium, 720px compact, overlay
   exclusivity, focus restoration, and minimum target/contrast rules.
5. **Interaction specification:** Define keyboard order, shortcuts, live-region
   messages, validation, destructive confirmations, and exact review semantics.
6. **Visual polish:** Only after the above, refine selective glass, agent
   identity, motion, and comfortable/compact density.

## Evidence limits

- Screenshots establish visible hierarchy, reflow, and many semantics, but do
  not prove WCAG compliance or native assistive-technology behavior.
- The browser preview uses deterministic demo data and cannot validate real
  repository, process, terminal, Git, persistence, security, or recovery work.
- Escape did not dismiss the command palette through two browser-input paths;
  this may be an app issue or a limitation of the capture harness, so native and
  automated verification is required before treating it as confirmed.
- The audit did not change runtime code. It records the design gate that should
  precede implementation.
